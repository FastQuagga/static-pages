Pipeline: HTML-верстка из raster mockup

Рабочий процесс для превращения PNG-мокапа в интерактивную HTML/CSS верстку. Главный принцип: pixel-perfect важен, но не ценой интерактивности. Изменяемые данные должны оставаться DOM-элементами.

На главную Открыть markdown-версию

Цель

Нужно получать HTML/CSS-верстку, визуально близкую к raster mockup, но пригодную для интеграции в игру: значения должны обновляться, кнопки нажиматься, шкалы менять прогресс, а статусы получать состояния.

  • Текст, числа, CTA, таблицы и состояния остаются live-DOM.
  • Raster допустим для иконок, гербов, портретов, сложных glyphs и декоративных фактур.
  • Bitmap-heavy варианты можно использовать для диагностики, но не как финальный UI.

1. Source of truth

Каждый блок начинается с original crop без масштабирования. Размеры фиксируются через sips -g pixelWidth -g pixelHeight, а координаты считаются от левого верхнего угла полного экрана.

2. Семантическая декомпозиция

Блок разбирается слоями: контейнер, фон, интерактивные группы, изменяемые значения, статические assets и состояния. Затем coarse block дробится на semantic sub-blocks и element layer.

  1. Coarse block: крупная зона или карточка.
  2. Semantic sub-blocks: смысловые группы внутри блока.
  3. Element layer: icon, value, label, divider, CTA.
  4. Raster candidate: только элементы, которые можно оставить картинками.
  5. Final DOM pass: возвращение к live-DOM для всех изменяемых частей.

3. Micro-crops

Micro-crops нужны, когда CSS/HTML плохо повторяет сложную форму: гербы, иконки, портреты, tactical field layer, sparkline или tiny glyph. Нельзя использовать micro-crop как финальный способ показать живой текст, проценты, суммы, таблицы или кнопку целиком.

4. Live-DOM верстка

layout.html каждого блока собирается как будущий UI: используются button, output, time, table, progress или обычные span с понятными data-* hooks.

Это может ухудшить raw pixel diff, потому что браузерный текст отличается от запеченного текста в PNG, но именно такой слой можно подключить к game state.

5. Сводная block-map.html

После декомпозиции полного мокапа обязательно собирается block-map.html: контрольная HTML-карта, где original, global-блоки и detail-блоки показаны на исходных координатах с тонкими цветными рамками и labels.

  • Crop-блоки вставляются без масштабирования.
  • Координаты должны совпадать с block-map-global.csv/json и block-map-detail.csv/json.
  • Карта должна помогать найти пропуски, дубли и смещения в нарезке экрана.
  • Если есть live-DOM блоки, добавляется Live-DOM Assembly Map с переключателем debug-рамок.
  • Отдельно нужен clean live-DOM screen без рамок, labels и debug-элементов.

6. Screenshot и pixel diff

Каждый блок рендерится через один и тот же Chrome headless pipeline с --force-device-scale-factor=1 и размером окна, равным размеру crop. Затем compare-png.mjs считает:

  • avgRgbDelta: средняя RGB-разница;
  • changedPixelsOver24: количество заметно измененных пикселей;
  • changedRatio: доля измененных пикселей;
  • maxRgbTripletDelta: максимальная RGB-разница одного пикселя.

7. Итерационная подгонка

Подгонка идет маленькими сериями: сначала внешний контейнер и фон, потом координаты групп, затем иконки, типографика, vertical rhythm, secondary labels и dividers. После каждой итерации сохраняется screenshot и diff-report.

8. Exact baseline

layout-exact.html показывает original crop как один raster слой. Он нужен не для интеграции, а чтобы понять нижнюю границу Chrome/color pipeline. Если exact тоже дает ненулевой diff, эту ошибку нельзя требовать от live-DOM верстки.

9. Финальная приемка

  • layout.html использует live-DOM для изменяемых значений.
  • Raster применяется только там, где он оправдан.
  • Side-by-side визуально читается как тот же UI-блок.
  • Для полного экрана есть block-map.html с original/global/detail картами, цветными рамками и координатами из block-map-*.csv/json.
  • Если блоки уже сверстаны, есть Live-DOM Assembly Map и clean live-DOM screen без debug-рамок.
  • Diff-метрики сохранены и объяснены.
  • Есть DOM hooks для будущей интеграции.
  • Пути валидны, CSS целый, служебных файлов нет.

Главный вывод

Лучший raw pixel diff не всегда означает лучший UI. Если pixel diff толкает к запеканию текста в PNG, нужно выбирать live-DOM и документировать tradeoff. Цель этого процесса - получить верстку, которую можно оживить, а не просто неподвижный скриншот.
